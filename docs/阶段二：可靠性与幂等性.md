## 1. 消息可靠性问题

### 1.1 消息丢失场景

```
场景1: Producer 发送失败
Producer → Broker (未收到) → 消息丢失

场景2: Broker 宕机
Producer → Broker → Leader 宕机 → 未同步到 Follower → 消息丢失

场景3: Consumer 消费后未提交 Offset
Consumer → 处理业务 → 异常退出 → 未提交 Offset → 消息重复消费
```

### 1.2 消息重复场景

```
场景1: Producer 重试导致
Producer → Broker (响应超时) → 重试发送 → Broker 收到两条

场景2: Consumer 提交 Offset 失败
Consumer → 处理业务 → 提交 Offset 失败 → 重新消费
```

## 2. 可靠性保障机制

### 2.1 生产者可靠性配置

```yaml
spring:
  kafka:
    producer:
      # acks 配置
      acks: all           # 等待所有 ISR 副本确认
      
      # 重试配置
      retries: 3         # 重试次数
      retry-backoff-ms: 1000  # 重试间隔
      
      # 幂等性配置
      properties:
        enable.idempotence: true     # 开启幂等性
        max.in.flight.requests.per.connection: 5  # 最大飞行请求数
        delivery.timeout.ms: 120000  # 消息发送超时
```

**acks 参数详解：**

| 值 | 含义 | 可靠性 | 性能 |
|----|------|--------|------|
| `0` | 不等待确认 | 低 | 高 |
| `1` | Leader 确认 | 中 | 中 |
| `all` | ISR 全部确认 | 高 | 低 |

### 2.2 生产者幂等性原理

开启幂等性后，Kafka 会为每个 Producer 分配一个唯一 ID（Producer ID），并为每条消息分配一个序列号（Sequence Number）：

```java
// 核心原理
- Producer ID (PID): 每个 Producer 唯一标识
- Sequence Number: 消息的序号
- Broker 会记录 (PID, Partition, Sequence Number)
- 收到重复消息时，Broker 会拒绝
```

### 2.3 消费者可靠性配置

```yaml
spring:
  kafka:
    consumer:
      # 关闭自动提交，改为手动提交
      enable-auto-commit: false
      auto-offset-reset: earliest  # 从最早的消息开始消费
      
    listener:
      ack-mode: manual  # 手动确认
```

## 3. 实战任务

### 3.1 任务一：实现消费者幂等性消费

在 `InventoryConsumerService` 中，我们使用 Redis 实现消息幂等性：

```java
@Autowired
private IdempotencyChecker idempotencyChecker;

@KafkaListener(topics = "order_events", groupId = "inventory-group")
public void consume(ConsumerRecord<String, OrderEvent> record, 
                    Acknowledgment acknowledgment) {
    String messageId = record.value().getMessageId();
    
    // 幂等性检查
    if (!idempotencyChecker.checkAndSave(messageId)) {
        log.warn("消息已处理，跳过: {}", messageId);
        acknowledgment.acknowledge();
        return;
    }
    
    // 处理业务逻辑
    processOrder(record.value());
    
    // 手动提交 Offset
    acknowledgment.acknowledge();
}
```

**幂等性检查工具类：**

```java
@Component
public class IdempotencyChecker {
    
    @Autowired
    private RedisTemplate<String, Object> redisTemplate;
    
    private static final String PREFIX = "msg:id:";
    private static final long EXPIRE_SECONDS = 86400; // 24小时
    
    public boolean checkAndSave(String messageId) {
        // NX: setIfAbsent，原子操作
        Boolean result = redisTemplate.opsForValue()
            .setIfAbsent(PREFIX + messageId, "1", 
                        Duration.ofSeconds(EXPIRE_SECONDS));
        return Boolean.TRUE.equals(result);
    }
}
```

### 3.2 任务二：实现分布式锁

在库存扣减场景中，需要防止并发导致的数据不一致：

```java
@Autowired
private DistributedLock distributedLock;

public boolean deductInventory(String productId, int quantity) {
    // 获取分布式锁
    String lockValue = distributedLock.acquireLock(productId);
    if (lockValue == null) {
        return false; // 获取锁失败
    }
    
    try {
        // 扣减库存
        return doDeductInventory(productId, quantity);
    } finally {
        // 释放锁
        distributedLock.releaseLock(productId, lockValue);
    }
}
```

**分布式锁实现：**

```java
@Component
public class DistributedLock {
    
    private static final String LOCK_PREFIX = "lock:";
    private static final Duration LOCK_TIMEOUT = Duration.ofSeconds(10);
    
    public String acquireLock(String key) {
        String lockValue = UUID.randomUUID().toString();
        Boolean acquired = redisTemplate.opsForValue()
            .setIfAbsent(LOCK_PREFIX + key, lockValue, LOCK_TIMEOUT);
        
        return Boolean.TRUE.equals(acquired) ? lockValue : null;
    }
    
    public boolean releaseLock(String key, String lockValue) {
        // 使用 Lua 脚本保证原子性
        String script = "if redis.call('get', KEYS[1]) == ARGV[1] " +
                       "then return redis.call('del', KEYS[1]) else return 0 end";
        redisTemplate.execute(new DefaultRedisScript<>(script), 
                             Collections.singletonList(LOCK_PREFIX + key), 
                             lockValue);
        return true;
    }
}
```

## 4. 完整处理流程

```
┌─────────────────────────────────────────────────────────────────┐
│                     消息处理流程                                  │
└─────────────────────────────────────────────────────────────────┘

1. 收到消息
   │
2. 幂等性检查 (Redis checkAndSave)
   ├── 已存在 → 跳过，直接 ACK
   └── 不存在 → 继续
   │
3. 获取分布式锁 (Redis)
   ├── 获取失败 → 不 ACK，等待重试
   └── 获取成功 → 继续
   │
4. 执行业务逻辑 (扣减库存)
   │
5. 释放分布式锁
   │
6. 手动提交 Offset (ACK)
```

## 7. 思考题

1. **幂等性和事务的区别？**
   - 幂等性：单条消息的去重
   - 事务：多条消息的原子性

2. **为什么需要分布式锁？**
   - 防止并发扣减导致库存为负数
   - 保证数据一致性

3. **手动提交 Offset 的风险？**
   - 处理完消息后提交失败，会导致重复消费
   - 提交前服务崩溃，也会导致重复消费
