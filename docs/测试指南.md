## 1. 快速开始

### 1.1 启动所有服务

```bash
# 进入项目目录
cd FlashStream

# 启动基础设施（包含监控工具）
docker-compose up -d

# 等待服务就绪（约30秒）
sleep 30

# 验证服务状态
docker-compose ps
```

### 1.2 启动应用服务

```bash
# 终端1: 启动订单服务
java -jar order-service/target/order-service.jar

# 终端2: 启动库存服务
java -jar inventory-service/target/inventory-service.jar

# 终端3: 启动通知服务
java -jar notification-service/target/notification-service.jar
```

## 2. 压测脚本

### 2.1 生产者压测

模拟秒杀期间的高并发下单流量：

```bash
# 基础用法（使用默认参数）
./scripts/producer-stress-test.sh

# 自定义参数
# 参数: [topic] [消息数] [消息大小] [吞吐量] [broker]
./scripts/producer-stress-test.sh order_events 100000 1024 -1 localhost:9091
```

**参数说明：**

| 参数 | 默认值 | 说明 |
|------|--------|------|
| topic | order_events | 测试的 Topic |
| num-records | 100000 | 发送的消息总数 |
| record-size | 1024 | 每条消息大小（字节） |
| throughput | -1 | 吞吐量限制（-1为无限制） |
| broker | localhost:9091 | Kafka 地址 |

**预期输出示例：**

```
100000 records sent, 50000.123456 records/sec (48.83 MB/sec), 234.56 ms avg latency, 123.45 ms 50th, 567.89 ms 99th, 890.12 ms 99.9th, 901.23 ms max latency.
```

### 2.2 消费者压测

模拟库存服务/通知服务的消费能力：

```bash
# 基础用法
./scripts/consumer-stress-test.sh

# 自定义参数
./scripts/consumer-stress-test.sh order_events 100000 localhost:9091 my-test-group
```

## 3. 指标查看

### 3.1 交互式菜单

```bash
./scripts/view-metrics.sh
```

功能选项：
1. 查看所有 Topic 列表
2. 查看 Topic 详情
3. 查看消费者组状态 (包含 LAG)
4. 查看指定消费者组详情
5. 查看 Broker 状态
6. 实时监控消息积压 (Watch LAG)
7. 查看服务端口映射

### 3.2 常用命令

```bash
# 查看 Topic 列表
docker exec kafka-1 kafka-topics --list --bootstrap-server localhost:9091

# 查看 Topic 详情
docker exec kafka-1 kafka-topics --describe --topic order_events --bootstrap-server localhost:9091

# 查看消费者组 LAG
docker exec kafka-1 kafka-consumer-groups \
    --bootstrap-server localhost:9091 \
    --group inventory-consumer-group \
    --describe
```

## 4. 业务验证

### 4.1 幂等性验证

验证重复消息不会导致库存重复扣减：

```bash
./scripts/test-idempotency.sh
```

**测试原理：**

1. 发送 5 次相同的订单请求（模拟网络抖动重试）
2. 检查 Redis 中的消息 ID 记录
3. 验证只有第一条消息被处理

**预期结果：**

```
发送次数: 5
唯一消息ID数: 1
✓ 幂等性验证通过！
只有第一条消息被处理，后续重复消息被正确拦截
```

### 4.2 Rebalance 测试

测试消费者组重平衡机制：

```bash
./scripts/test-rebalance.sh
```

**测试步骤：**

1. 查看当前消费者组状态
2. 发送测试消息
3. 启动新的消费者实例
4. 观察 Rebalance 过程
5. 停止消费者实例
6. 再次观察 Rebalance

**触发 Rebalance 的方式：**

```bash
# 方式1: 新消费者加入
docker-compose up -d inventory-service-2

# 方式2: 消费者离开
docker-compose stop inventory-service

# 方式3: 重置 offset
docker exec kafka-1 kafka-consumer-groups \
    --bootstrap-server localhost:9091 \
    --group inventory-consumer-group \
    --reset-offsets \
    --to-earliest \
    --topic order_events \
    --execute
```

## 6. 关键指标解读

### 6.1 吞吐量 (Throughput)

| 指标 | 说明 | 秒杀场景参考值 |
|------|------|----------------|
| records/sec | 每秒发送/接收的消息数 | > 10,000 |
| MB/sec | 每秒传输的数据量 | > 10 MB/s |

### 6.2 延迟 (Latency)

| 指标 | 说明 | 秒杀场景参考值 |
|------|------|----------------|
| avg latency | 平均延迟 | < 10 ms |
| 50th | 中位数延迟 | < 5 ms |
| 99th | 99分位延迟 | < 50 ms |
| max | 最大延迟 | < 200 ms |

### 6.3 消息积压 (Lag)

**最关键指标**，表示消费者处理速度落后于生产者的程度：

| Lag 值 | 状态 | 建议 |
|--------|------|------|
| 0 | 正常 | 无需处理 |
| 1-100 | 轻微积压 | 关注 |
| 100-1000 | 中度积压 | 增加消费者 |
| > 1000 | 严重积压 | 紧急处理 |

**查看 LAG 命令：**

```bash
docker exec kafka-1 kafka-consumer-groups \
    --bootstrap-server localhost:9091 \
    --group inventory-consumer-group \
    --describe
```

**输出示例：**

```
GROUP                          TOPIC           PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
inventory-consumer-group      order_events    0          1000            1500            500
```

## 8. 常见问题

### Q1: 压测时消息发送失败

**可能原因：**
- Kafka 集群未就绪
- Topic 不存在

**解决方案：**
```bash
# 等待 Kafka 就绪
docker logs -f kafka-1 | grep "started"

# 手动创建 Topic
docker exec kafka-1 kafka-topics --create \
    --topic order_events \
    --partitions 6 \
    --replication-factor 3 \
    --bootstrap-server localhost:9091
```

### Q2: LAG 持续增长

**可能原因：**
- 消费者处理速度慢
- 消费者实例不足

**解决方案：**
```bash
# 增加消费者实例
docker-compose up -d inventory-service-2 inventory-service-3

# 优化消费逻辑（批量处理）
```

### Q3: Rebalance 频繁触发

**可能原因：**
- 消费者心跳超时
- 处理时间过长

**解决方案：**
```bash
# 增加心跳超时配置
spring:
  kafka:
    consumer:
      heartbeat-interval-ms: 3000
      session-timeout-ms: 45000
      max-poll-interval-ms: 300000
```
