## 1. Kafka 基本概念

### 1.1 核心术语

| 术语 | 说明 |
|------|------|
| **Broker** | Kafka 服务实例，一个 Kafka 集群由多个 Broker 组成 |
| **Topic** | 消息的主题分类，类似数据库的表 |
| **Partition** | Topic 的分区，实现消息的并行处理和水平扩展 |
| **Replica** | 分区的副本，实现高可用 |
| **ISR** | In-Sync Replicas，与 Leader 保持同步的副本集合 |
| **Consumer Group** | 消费者组，实现消息的负载均衡消费 |
| **Offset** | 消息在分区中的位置索引 |

### 1.2 消息发送流程

```
Producer → Broker (Leader) → ISR (同步副本) → ACK
```

## 2. 实战任务

### 2.1 任务一：使用 KafkaAdminClient 创建 Topic

在 `OrderServiceApplication` 中，我们已经通过 `@Bean` 自动创建了 Topic。但在本任务中，你需要**手动**使用 KafkaAdminClient 来创建。

**创建 `KafkaAdminClientConfig.java`：**

```java
package com.flashstream.order.config;

import com.flashstream.common.constant.KafkaConstants;
import org.apache.kafka.clients.admin.AdminClient;
import org.apache.kafka.clients.admin.AdminClientConfig;
import org.apache.kafka.clients.admin.NewTopic;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.kafka.core.AdminClientTemplate;

import java.util.HashMap;
import java.util.Map;

@Configuration
public class KafkaAdminClientConfig {

    @Bean
    public AdminClient adminClient() {
        Map<String, Object> configs = new HashMap<>();
        configs.put(AdminClientConfig.BOOTSTRAP_SERVERS_CONFIG, 
                KafkaConstants.BOOTSTRAP_SERVERS);
        return AdminClient.create(configs);
    }

    @Bean
    public NewTopic orderEventsTopicManual() {
        return new NewTopic(
                KafkaConstants.ORDER_EVENTS_TOPIC,
                KafkaConstants.PARTITION_COUNT,
                (short) KafkaConstants.REPLICATION_FACTOR
        );
    }
}
```

**手动创建 Topic 的代码：**

```java
@Autowired
private AdminClient adminClient;

public void createTopic(String topicName, int partitions, short replicationFactor) {
    NewTopic newTopic = new NewTopic(topicName, partitions, replicationFactor);
    CreateTopicsResult result = adminClient.createTopics(Collections.singletonList(newTopic));
    result.all().get(); // 等待创建完成
}
```

### 2.2 任务二：实现基础 Producer

在 `OrderProducerService` 中，我们已经实现了 Producer。以下是核心代码解析：

```java
@Autowired
private KafkaTemplate<String, Object> kafkaTemplate;

public CompletableFuture<SendResult<String, Object>> sendMessage(String topic, 
                                                                  String key, 
                                                                  Object message) {
    return kafkaTemplate.send(topic, key, message);
}
```

**关键配置说明：**

```yaml
spring:
  kafka:
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
      acks: all          # 等待所有副本确认
      retries: 3         # 重试次数
```

## 4. 分区策略详解

### 4.1 默认分区策略

Kafka 默认使用以下分区策略：
1. 如果指定了分区，直接发送到指定分区
2. 如果指定了 Key，按 Key 的 hash 值计算分区
3. 如果都没有指定，使用轮询策略

### 4.2 我们的分区策略

在订单服务中，我们使用 `userId` 作为分区键：

```java
String partitionKey = request.getUserId();
kafkaTemplate.send(topic, partitionKey, orderEvent);
```

**这样可以确保：**
- 同一用户的订单消息发送到同一个分区
- 保证消息顺序性（同一用户的订单按顺序处理）

## 5. 思考题

1. **为什么需要分区？**
   - 实现消息的并行处理
   - 提高系统的吞吐量
   - 实现消息的负载均衡

2. **分区数与消费者数的关系？**
   - 分区数 >= 消费者数，才能充分利用消费者
   - 分区数过多会增加 Zookeeper 负担

3. **如何选择分区键？**
   - 选择业务上有关联性的字段
   - 避免数据倾斜（如使用 UUID 作为分区键）
