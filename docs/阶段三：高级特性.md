## 1. 消息积压问题

### 1.1 积压原因

```
场景1: 消费者处理能力不足
Producer (1000/s) → Kafka → Consumer (100/s) → 积压

场景2: 消费者服务宕机
Producer → Kafka (积压) → Consumer 恢复 → 短时间内无法处理完

场景3: 业务逻辑复杂
Consumer 处理一条消息需要 10s → 积压
```

### 1.2 解决方案

| 方案 | 适用场景 | 说明 |
|------|----------|------|
| 增加分区 | 消费者能跟上 | 提高并行度 |
| 增加消费者 | 消费者是瓶颈 | 扩展消费能力 |
| 批量消费 | IO 密集型 | 减少网络开销 |
| 异步处理 | 非实时 | 消息放入缓冲 |

## 2. 动态分区

### 2.1 增加分区数

```bash
# 进入 Kafka 容器
docker exec -it kafka-1 bash

# 增加分区数（只能增加，不能减少）
kafka-topics --alter \
  --topic order_events \
  --partitions 12 \
  --bootstrap-server localhost:9091
```

### 2.2 分区再分配

当消费者组发生变化时，Kafka 会触发 Rebalance：

```bash
# 查看当前消费者组状态
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9091 \
  --group inventory-consumer-group \
  --describe
```

### 2.3 Rebalance 策略

Kafka 提供三种分区分配策略：

| 策略 | 说明 | 配置 |
|------|------|------|
| Range | 按主题逐个分配 | `partition.assignment.strategy=org.apache.kafka.clients.consumer.RangeAssignor` |
| RoundRobin | 轮询分配 | `partition.assignment.strategy=org.apache.kafka.clients.consumer.RoundRobinAssignor` |
| Sticky | 粘性分配（推荐） | `partition.assignment.strategy=org.apache.kafka.clients.consumer.CooperativeStickyAssignor` |

**配置方式：**

```yaml
spring:
  kafka:
    consumer:
      properties:
        partition.assignment.strategy: org.apache.kafka.clients.consumer.CooperativeStickyAssignor
```

## 3. 批量消费

### 3.1 消费者批量配置

```yaml
spring:
  kafka:
    consumer:
      # 批量配置
      max-poll-records: 500          # 每次 poll 最大记录数
      fetch-min-size: 1048576        # 最小获取字节数 (1MB)
      fetch-max-wait-ms: 500          # 最大等待时间
      max-poll-interval-ms: 300000    # 消费者处理超时
      
    listener:
      type: batch                     # 批量监听
      ack-mode: manual
```

### 3.2 批量消费代码

```java
@KafkaListener(topics = "order_events", groupId = "inventory-group")
public void consumeBatch(List<ConsumerRecord<String, OrderEvent>> records,
                         Acknowledgment acknowledgment) {
    
    log.info("收到批量消息: {} 条", records.size());
    
    for (ConsumerRecord<String, OrderEvent> record : records) {
        try {
            processOrder(record.value());
        } catch (Exception e) {
            log.error("处理消息失败: {}", record.value().getOrderId(), e);
        }
    }
    
    // 批量提交
    acknowledgment.acknowledge();
}
```

### 3.3 生产者批量配置

```yaml
spring:
  kafka:
    producer:
      # 批量发送配置
      batch-size: 16384               # 批量大小 (16KB)
      buffer-memory: 33554432         # 缓冲区大小 (32MB)
      compression-type: snappy        # 压缩类型
      linger-ms: 10                   # 等待时间
```

## 4. 性能优化

### 4.1 压缩配置

Kafka 支持多种压缩算法：

| 算法 | 压缩比 | CPU 使用 | 适用场景 |
|------|--------|----------|----------|
| gzip | 高 | 高 | 网络带宽受限 |
| snappy | 中 | 中 | 平衡场景（默认） |
| lz4 | 中 | 低 | 低延迟场景 |
| zstd | 高 | 中 | 高压缩比场景 |

**配置：**

```yaml
spring:
  kafka:
    producer:
      compression-type: snappy
```

### 4.2 顺序保证

**方案一：单分区顺序**

```java
// 使用相同的分区键，保证同一用户的订单在同一分区
kafkaTemplate.send(topic, userId, message);
```

**方案二：事务保证**

```java
@Transactional
public void sendOrderAndNotify(OrderEvent order) {
    // 发送订单消息
    kafkaTemplate.send("order_events", order.getUserId(), order);
    
    // 发送通知消息（同一事务）
    kafkaTemplate.send("notification_events", order.getUserId(), notification);
    
    // 提交事务
    kafkaTemplate.flush();
}
```

**Kafka 事务配置：**

```yaml
spring:
  kafka:
    producer:
      transaction-id-prefix: flashstream-tx-
      properties:
        enable.idempotence: true
```

### 5.2 任务二：配置批量消费

修改 `InventoryConsumerService`：

```java
@KafkaListener(
    topics = KafkaConstants.ORDER_EVENTS_TOPIC,
    groupId = KafkaConstants.INVENTORY_CONSUMER_GROUP,
    containerFactory = "batchKafkaListenerContainerFactory"
)
public void consumeBatch(List<ConsumerRecord<String, OrderEvent>> records,
                        Acknowledgment acknowledgment) {
    
    log.info("收到批量消息: {} 条", records.size());
    
    // 批量处理
    List<String> messageIds = records.stream()
        .map(r -> r.value().getMessageId())
        .collect(Collectors.toList());
    
    log.info("消息IDs: {}", messageIds);
    
    // 批量提交
    acknowledgment.acknowledge();
}
```

## 8. 完整项目总结

### 8.1 技术架构

```
┌──────────────────────────────────────────────────────────────────┐
│                         FlashStream 架构                          │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  [前端] → [Order Service] → [Kafka Cluster]                     │
│                    │                    │                        │
│                    │                    ├─→ [Inventory Service] │
│                    │                    │         ↓              │
│                    │                    │    分布式锁 + 幂等    │
│                    │                    │                        │
│                    │                    └─→ [Notification]      │
│                    │                            ↓                │
│                    │                        短信通知              │
│                    │                                                 │
│                    └─→ [Redis] 幂等校验 + 分布式锁               │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

### 8.2 Kafka 核心知识点

1. **消息可靠性**
   - acks=all + 重试
   - 幂等性生产
   - 手动提交 Offset

2. **消息顺序性**
   - 同一分区保证顺序
   - 分区键选择

3. **性能优化**
   - 批量发送/消费
   - 压缩
   - 增加分区

## 10. 常见问题

**Q: 消息积压怎么办？**
A: 增加消费者实例、增加分区、优化消费逻辑

**Q: 如何保证消息顺序？**
A: 使用相同的分区键，确保同一业务的消息发往同一分区

**Q: Rebalance 期间有什么影响？**
A: 消费者会暂停消费，直到 Rebalance 完成
